Create human-reviewable implementation plans as designed HTML documents with clear structure, diagrams, and acceptance criteria before starting substantial work.

Use this workflow when a user asks for a plan, spec, feature design, implementation approach, handoff, architecture proposal, or anything that needs human review before code changes.

Core rule:
- For any non-trivial plan, create or update an HTML plan document before implementation.
- Do not treat a Markdown wall of text as sufficient when the plan is meant for human review.
- Make the HTML useful to inspect visually in a browser, not merely valid HTML.

Success looks like:
- The reviewer can open one local HTML file and understand the proposal quickly.
- The document makes sequencing, data flow, ownership, risks, and open decisions obvious.
- The plan has concrete acceptance criteria and verification steps.
- The design makes review easier: clear sections, readable typography, visual hierarchy, diagrams, callouts, and compact tables where helpful.

Planning workflow:
1. Identify the target project and likely output location.
   - Prefer an existing `plans/` directory when present.
   - Otherwise create the plan near the relevant project docs.
   - Use a descriptive kebab-case filename ending in `-spec.html` or `-plan.html`.
2. Read enough existing code/docs to make the plan concrete.
3. Create the HTML plan document.
4. Tell the user the exact local path and, when useful, a `file://` URL.
5. Wait for human approval before starting implementation unless the user explicitly asks to continue.

HTML document requirements:
- Use a complete standalone HTML file with embedded CSS.
- Keep it dependency-free unless the repo already has a clear local convention for plan docs.
- Include:
  - Title and concise subtitle
  - Problem or goal
  - Proposed behavior or architecture
  - Implementation tasks in reviewable chunks
  - Data model/API/contract changes when relevant
  - Sequencing or rollout notes
  - Risks and edge cases
  - Acceptance criteria
  - Verification plan
  - Open decisions
- Use code blocks for schemas, payloads, commands, or exact contracts.
- Use tables for comparisons, route maps, state machines, or file ownership when they improve scanning.

Design requirements:
- Make the doc feel intentionally designed, not generated by default browser styles.
- Use restrained typography, spacing, color, and borders.
- Keep sections visually distinct and easy to scan.
- Add small diagrams where they clarify the plan:
  - Flow diagrams for request/event/scheduler paths
  - Sequence diagrams for multi-actor interactions
  - State diagrams for lifecycle changes
  - Box-and-arrow diagrams for data movement
- Inline SVG diagrams are preferred because they keep the plan as one portable file.
- Label diagrams directly; do not rely on surrounding prose to explain every node.
- Use callouts for important constraints, reviewer warnings, and non-obvious decisions.
- Avoid decorative filler. Every visual element should make the plan easier to review.

Tone and content rules:
- Be concrete and implementation-aware.
- State assumptions explicitly.
- Separate decided behavior from open decisions.
- Do not hide uncertainty in confident prose.
- Keep sections short enough that a reviewer can skim, but include enough specifics that implementation can proceed after approval.
- Use plain language for product behavior and precise language for technical contracts.

Review gate:
- If the user says something like "continue", "build it", or "implement it" after seeing the plan, proceed from the approved HTML plan.
- If the user asks for changes to the plan, update the HTML first and point them back to it.
- If implementation reveals the approved plan is wrong, update the HTML plan or clearly explain the divergence before continuing.

Quality check before handing off:
- Open/read the generated file enough to catch malformed markup.
- Confirm the file exists and has non-empty content.
- If a browser or screenshot tool is available and the visual design is important, inspect it visually.
- Make sure text is readable on mobile-width and desktop-width layouts.

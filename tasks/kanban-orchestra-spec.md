# Kanban Orchestra Spec 2.0

This document is the canonical behavior spec for Kanban Orchestra.

## Scope

- One work repo at a time.
- One orchestrator instance per repo.
- One active task at a time.
- One task maps to one eventual landed commit.
- The task database is the durable source of truth.

## Repo Model

Two roles participate:

- Work Repo: the repo where code changes happen
- Orchestra Repo: the repo that contains shared orchestration scripts and prompts

The two roles may be the same checkout when Orchestra is working on itself.

The work repo contains the live database:

```text
<work-repo>/
  kanban-orchestra.db
  kanban-orchestra.sql
```

The Orchestra repo is resolved via `ORCHESTRA_DIR` and contains:

```text
$ORCHESTRA_DIR/
  kanban-orchestra/
    scripts/
    prompts/
```

Rules:

- All file edits, builds, git operations, and task execution happen in the launched work repo.
- `ORCHESTRA_DIR` must point to an Orchestra checkout root.
- Use launched-repo paths as the write location. When the launched repo is also
  `$ORCHESTRA_DIR`, Orchestra source files are valid task targets.
- The orchestrator assumes exclusive repo access in the current worktree.

## Core Concepts

### Task Types

A task is the main durable unit of work. Its public type controls which
workflow drives it.

- It has a durable database identity: `id`
- It has a workflow state: `status`, `next_step`, `review_round`
- Commit tasks may eventually have a landed git identity: `commit_hash`
- A commit task's landed commit message must end with `Task <id> (<attribution>)` — the canonical footer generated by `task get-commit-footer <id>`
- Pull request tasks manage PR metadata and do not land commits.
- Other tasks handle non-standard work and require durable task-comment evidence instead of a commit or PR.

### Supertask

A supertask is a planning container.

- It creates and sequences child tasks.
- It never lands a commit.
- Its children remain the executable units of work.

### Sticky Coder

The sticky coder is the agent assigned to `commit-make`.

- Default: `sonnet`
- Stored on the task as `coder_agent`
- Reused for the life of the task unless a human changes it

### Reviewer

The reviewer is the agent assigned to `commit-review` for a task.

- Default: `codex`
- Stored on the task as `reviewer_agent`
- If the field is missing or null in an older database row, runtime code falls back to `DEFAULT_REVIEWER`
- Review is sequential
- Review produces exactly one approval or rejection for the current round

### Default Role Overrides

Default agent roles can be overridden at runtime with Unix environment variables.
If the variable is unset, empty, whitespace-only, or names an unknown agent, the
hard-coded default is used.

| Environment variable              | Constant overridden    | Hard-coded default |
|-----------------------------------|------------------------|--------------------|
| `ORCHESTRA_DEFAULT_SUPER_PLANNER` | `DEFAULT_SUPER_PLANNER`| `opus`             |
| `ORCHESTRA_DEFAULT_SUPER_REVIEWER`| `DEFAULT_SUPER_REVIEWER`| `codex`           |
| `ORCHESTRA_DEFAULT_PLANNER`       | `DEFAULT_PLANNER`      | `sonnet`           |
| `ORCHESTRA_DEFAULT_PLAN_REVIEWER` | `DEFAULT_PLAN_REVIEWER`| `codex`            |
| `ORCHESTRA_DEFAULT_CODER`         | `DEFAULT_CODER`        | `sonnet`           |
| `ORCHESTRA_DEFAULT_REVIEWER`      | `DEFAULT_REVIEWER`     | `codex`            |

### Review Round

`review_round` is the durable counter for review attempts.

- Initial value: `0`
- Incremented after a rejection
- Also incremented when a review round is abandoned due to interruption or stale recovery

## System Invariants

These rules define the system.

- One task maps to one eventual landed commit.
- `id` is the durable task identity. `commit_hash` is not.
- Only the orchestrator advances workflow state during normal operation.
- Review outcomes are stored as comments; the orchestrator derives `last_review_decision`.
- `done` means the task's commit was finalized and submitted.
- A task must have a branch before it can be set to `ready`.
- A task should have a meaningful `next_step` before it is set to `ready`.
- Tasks on `master` or `main` are disabled by default unless the work repo
  explicitly opts in with `ALLOW_TASKS_ON_MASTER` in `AGENTS.md`.
- A dirty worktree blocks pickup of a normal `commit-make` task.
- The orchestrator processes one pinned task at a time through its full local loop.
- No other task may leapfrog between phases while a pinned task is still active.

## Database Compatibility Policy

- It is never committed to git.
- It does not need to be preserved across schema changes.
- There is no automatic migration of old DB files to new schemas.

When a schema change requires a table recreation:

1. Delete `kanban-orchestra.db` (and its `-shm`/`-wal` sidecar files if present).
2. Reopen: `db.connect()` will bootstrap a fresh database automatically.
3. Alternatively, restore from a current `kanban-orchestra.sql` dump using `task restore`.

`db.connect()` checks for incompatible schemas on open and raises a `RuntimeError`
with recovery instructions if the DB is stale.

The `kanban-orchestra.sql` dump file is the restore source. If a dump was produced
by an older schema, regenerate it from a current database.

## Storage Model

### Database Files

- `kanban-orchestra.db`: live source of truth
- `kanban-orchestra.sql`: optional tracked SQL dump of the live DB
- `.kanban-orchestra/orchestrator.log`: persisted orchestrator stdout for operator and dashboard use

### Tables

#### `tasks`

Primary durable workflow state.

Important columns:

- `id`
- `title`
- `description`
- `status`
- `next_step`
- `branch`
- `commit_hash`
- `stash_ref`
- `coder_agent`
- `reviewer_agent`
- `review_round`
- `last_review_decision`
- `ready_at`
- `kind`
- `parent_task_id`
- `sequence_index`
- `commit_plan`
- `allow_when_blocked`

#### `task_skips`

Normalized storage for step-skipping flags.

Columns:

- `task_id`: foreign key to `tasks(id)`
- `step`: one of `commit-plan`, `commit-plan-review`, `commit-review`, `commit-review-supertask`, `pull-request-review`, `other-review`

A task is skipped for a given step if a row exists in this table for that `task_id` and `step`.

Enumerations currently implemented:

- `status`: `none`, `ready`, `running`, `done`, `blocked`, `pending_subtasks`
- `next_step`: `commit-make`, `commit-review`, `commit-make-supertask`, `commit-review-supertask`, `commit-plan`, `commit-plan-review`, `pull-request-make`, `pull-request-review`, `other-make`, `other-review`, `none`
- `kind`: storage column for task type; current values are `commit`, `supertask`, `pull_request`, and `other`. Legacy `task` rows are accepted as commit tasks.
- `last_review_decision`: `none`, `approve`, `reject`

#### `comments`

Permanent task history.

Important comment kinds:

- `comment`
- `approval`
- `rejection`
- `commit-message`
- `validation`
- `plan-approval`
- `plan-rejection`

Comments are never purged by normal operation.

#### `run_log`

Ephemeral operational log.

- Stores task-scoped orchestrator and agent progress messages
- Ordered newest-first in the CLI
- May be purged

#### `orchestrator_runtime`

Single-row live runtime state for operational visibility.

Important fields:

- `status`
- `pid`
- `started_at`
- `last_heartbeat_at`
- `current_task_id`
- `current_step`
- `current_branch`
- `review_round`
- `active_agents`
- `status_message`

Runtime status values:

- `idle`
- `running`
- `starting`
- `stopping`
- `stopped`
- `hard-break`
- `error`

### Runtime Files

- `kanban-orchestra.lock`: repo-scoped singleton lock for the real orchestrator
  process. Its metadata records the process PID, launch repo root, repo label,
  database path, runtime directory, and lock path so local agents can identify
  the orchestrator for the current repo without process hunting.
- `.kanban-orchestra/orchestrator.log`: append-only stdout log for each orchestrator process
- `.kanban-orchestra/dashboard.json`: repo-scoped metadata for the dashboard
  owned by the orchestrator instance, including PID, host, port, and URL.
- `.kanban-orchestra/artifacts/`: filesystem-backed run artifacts such as transcripts
- `.kanban-orchestra/active-agent-processes.json`: transient process-group
  metadata for active agent children, maintained by the active-agent runtime
  helpers

## Task States

### `status`

- `none`: task exists but is parked
- `ready`: queued for orchestration
- `running`: currently being processed by the orchestrator
- `done`: landed and submitted
- `blocked`: requires human intervention
- `pending_subtasks`: approved supertask plan with child execution in progress

### `next_step`

- `commit-make`: coder builds or finalizes a commit task
- `commit-review`: reviewer inspects the staged diff for a commit task
- `commit-make-supertask`: coder plans a supertask
- `commit-review-supertask`: reviewer reviews a supertask plan
- `commit-plan`: coder drafts an implementation plan for a normal task
- `commit-plan-review`: reviewer approves or rejects the drafted plan
- `none`: no pending step

## Git Contract

Kanban Orchestra uses git as the execution workspace, not as the primary state
machine.

Rules:

- During normal Path A `commit-make`, the coder stages the candidate diff with `git add .`.
- `commit-review` reviews `git diff --cached`.
- The real git commit is not created until approval.
- Finalization uses a plain `git commit`, never amend.
- The orchestrator records the resulting `HEAD` as `commit_hash`.
- If `commit-make` becomes blocked after local changes exist, the orchestrator stages and stashes WIP, records `stash_ref`, and leaves a durable comment.

## Deferred Full-Build Validation (SKIP_BUILD_UNTIL_APPROVED)

Some repos — particularly those with expensive builds (e.g. iOS apps) — may defer
full-build validation until after initial code review rather than running it on every
`commit-make` iteration.

### Policy Marker

Repos opt in by adding this exact line to their root `AGENTS.md`:

```
SKIP_BUILD_UNTIL_APPROVED
```

The marker must appear as a standalone line (leading/trailing whitespace is ignored).
Freeform prose containing the string does not qualify.

### Behavior on Path A (`last_review_decision != approve`)

When the marker is present:

- The coder **skips the full build** on Path A.
- The coder **must** record a `validation` comment explicitly stating that the full
  build was intentionally deferred by `SKIP_BUILD_UNTIL_APPROVED` policy.
- Reviewers see this context in the `## Reviewer Handoff` section and must **not**
  reject solely because a full-build result is absent.

### Behavior on Path B (`last_review_decision == approve`)

When the marker is present, the coder runs the full build **before** creating the commit.

#### Clean build — finalize normally

If the build passes without modifying any staged files, the coder creates the commit
as usual. Finalization proceeds normally.

#### Build changes the diff — re-enter review

If the build modifies files (e.g. auto-generated artifacts, formatter output, required
fixes), the coder:

1. Stages all changes with `git add .`.
2. Records a `deferred-build-changed` comment describing what changed.
3. Exits **without** creating a commit.

The orchestrator detects the `deferred-build-changed` signal and re-enters
`commit-review` at the next review round. The reviewer then evaluates the
build-modified diff before it lands. This ensures no unreviewed post-approval
changes reach the repo.

### State Transitions for Deferred Build Path B

#### Deferred build passes cleanly

Same as normal Path B finalization:
- `commit_hash <- HEAD`
- `status -> done`
- `next_step -> none`

#### Deferred build changes diff

- `status -> ready`
- `next_step -> commit-review`
- `review_round += 1`
- `last_review_decision -> none`

On the subsequent Path B after that review approves, the build should be clean
(since the modified diff was already incorporated) and finalization proceeds normally.

### Source of Truth

`repo_policy.read_skip_build_until_approved(repo_root)` is the single parsing
function. All orchestrator and prompt behavior is driven by this function's output.

## Master/Main Task Opt-In (ALLOW_TASKS_ON_MASTER)

Fresh repos should not accidentally queue or run Kanban work on their primary
branch. Repos that intentionally operate directly on `master` or `main` opt in by
adding this exact standalone line to their root `AGENTS.md`:

```
ALLOW_TASKS_ON_MASTER
```

The marker must appear as a standalone line (leading/trailing whitespace is ignored).
Freeform prose containing the string, including colon variants such as
`ALLOW_TASKS_ON_MASTER: true`, does not qualify.

When the marker is absent:

- `task add --branch master` and `task add --branch main` fail.
- `task set --branch master` and `task set --branch main` fail.
- `task set --status ready` fails when the resulting branch is `master` or
  `main`, including tasks that already carry that branch.
- The orchestrator defensively blocks any existing ready task on `master` or
  `main` before launching an agent and records a durable comment explaining the
  policy and remediation.

When the marker is present, these operations are allowed and task prompt context
surfaces `allow_tasks_on_master: yes` for tasks on `master` or `main`.

`repo_policy.read_allow_tasks_on_master(repo_root)` is the single parsing
function. CLI and orchestrator behavior is driven by this function's output.

## CLI Contract

`task.py` is the canonical mutation and inspection interface for humans and agents.

### Read Commands

```bash
task list [--status <status>] [--next-step <step>] [--branch <branch>] [--page <n>]
task show <task-id>
task show-comments <task-id>
task show-run-log <task-id>
```

### Mutation Commands

```bash
task add "<title>" [--type <commit|pull_request|supertask|other>] [--description "<description-as-markdown>"] [--branch <branch>] [--coder-agent <agent>] [--reviewer-agent <agent>] [--kind <legacy-kind>] [--parent <task-id>] [--sequence-index <n>] [--skip <step>] [--allow-when-blocked]
task set <task-id> [--title "<title>"] [--description "<description-as-markdown>"] [--status <status>] [--next-step <step>] [--branch <branch>] [--commit <hash>] [--stash-ref <ref>] [--coder-agent <agent>] [--reviewer-agent <agent>] [--review-round <n>] [--last-review-decision <decision>] [--sequence-index <n>] [--commit-plan "<text>"] [--allow-when-blocked <bool>] [--add-skip <step> | --remove-skip <step>]
task comment <task-id> [<message> | --message-stdin] [--comment | --approval | --rejection | --commit-message | --validation | --plan-approval | --plan-rejection] [--author <name>] [--review-round <n>]
task log <task-id> "<message>"
task delete <task-id>
task purge [--before <date> | --days <n>]
task dump
task restore
```

### CLI Rules

- `task add` defaults to `type=commit`, `coder_agent=sonnet`, and `reviewer_agent=codex`. `--kind task` remains a legacy alias for `--type commit`.
- `task set --reviewer-agent <agent>` changes the configured code reviewer for future `commit-review` runs.
- `allow_when_blocked` defaults to `false`.
- Task descriptions are Markdown source. Agents creating or editing tasks should
  use Markdown structure such as headings, bullets, and code spans where useful.
- Supertasks start with `next_step=commit-make-supertask`.
- Commit tasks default to storing a `commit-plan` skip and start with `next_step=commit-make`. `commit-plan-review` is inferred from the `commit-plan` skip, so default task creation does not need to store both skips. Additional `--skip` values are unioned with this default.
- Pull request tasks start with `next_step=pull-request-make`; `pull-request-review` is the only valid pull request skip.
- Other tasks start with `next_step=other-make`; `other-review` is the only valid other-task skip.
- Parented child tasks inherit the parent branch.
- `--branch` is not allowed when creating or directly editing a child task.
- Setting `status=ready` requires a branch.
- `master` and `main` branches require `ALLOW_TASKS_ON_MASTER` as a standalone
  line in the work repo `AGENTS.md` before tasks can be added, retargeted, or
  set to `ready` on those branches.
- In agent or non-interactive mode, missing branch on `status=ready` is an error.
- `task comment --message-stdin` reads the full message from stdin and is the preferred form for multi-line or shell-sensitive text.
- `task delete` only works for tasks still in `status=none`.
- Approval and rejection comments require both `--author` and `--review-round`.
- Approval and rejection writes are rejected if the provided round is stale.
- Plan approval and rejection comments (`--plan-approval`, `--plan-rejection`) require `--author` but do not use `--review-round`.

## Commit Task Workflow

Commit tasks use a two-step loop:

1. `commit-make`
2. `commit-review`

### `commit-make`

Agent: sticky coder

The coder must branch on `stash_ref` and `last_review_decision`.

#### Path C: `stash_ref` is set

This path takes precedence.

- Attempt `git stash pop <stash_ref>`.
- If clean, clear `stash_ref` and continue.
- If conflicted, drop the stash, clear `stash_ref`, and rebuild from task history.
- If the stash is missing, clear `stash_ref`, leave a durable comment, and rebuild.

#### Path A: `last_review_decision != approve`

This is the build or rework path.

Required behavior:

- Read prior comments, especially rejections and human guidance.
- Implement the requested change.
- Run the repo-defined validation command when the task requires it. If the repo has
  `SKIP_BUILD_UNTIL_APPROVED` in `AGENTS.md`, skip the full build on Path A and
  record a `validation` comment stating the deferral explicitly.
- Record validation results as a `validation` comment for the current round.
- Stage the full candidate diff with `git add .`.
- Record a fresh `commit-message` comment during this run.
- Do not create the actual git commit.

Orchestrator enforcement:

- If Path A exits successfully without a new `commit-message` comment, the run is treated as failure.

#### Path B: `last_review_decision == approve`

This is the finalization path.

Required behavior:

- Read the current-round approval comment and any same-round review notes before
  committing.
- Treat approval as permission to land, not as a reason to ignore reviewer
  guidance. The coder may make small, directly reviewer-requested final edits
  that are clearly within the approved scope, then stage them before committing.
- Do not make broader improvements, refactors, or opportunistic fixes on Path B.
  If a requested edit is non-trivial, a new issue appears, or scope is unclear,
  the coder must avoid committing and leave a durable comment so the task can
  return to review or human triage.
- Read the latest `commit-message` comment.
- Ensure the canonical footer `Task <id> (<attribution>)` is present. Run `task get-commit-footer <id>` to produce it.
- If the repo has `SKIP_BUILD_UNTIL_APPROVED`, run the full build now. If the
  build changes the staged diff, record a `deferred-build-changed` comment and exit
  without committing; the orchestrator re-enters review. If the build is clean,
  proceed to commit.
- Create the real commit with plain `git commit`.

### `commit-review`

Agent: the task's `reviewer_agent`, falling back to `DEFAULT_REVIEWER` for old/null rows.

Required behavior:

- Inspect `git diff --cached`.
- Read the current task context, prior comments, latest commit message, and latest validation summary for the current round.
- Record exactly one `approval` or `rejection` comment for the current round.
- Do not edit files.
- Do not mutate task state directly.

Review aggregation rules:

- The orchestrator looks only at reviewer comments for the current round.
- A missing reviewer decision after a successful reviewer process is treated as rejection.
- The reviewer comment itself does not set `last_review_decision`; the orchestrator does.

### Commit Task State Transitions

#### On successful Path A `commit-make`

- `status -> ready`
- `next_step -> commit-review`
- `last_review_decision -> none`

#### On review approval

- `status -> ready`
- `next_step -> commit-make`
- `last_review_decision -> approve`

#### On review rejection

- `status -> ready`
- `next_step -> commit-make`
- `review_round += 1`
- `last_review_decision -> reject`

#### On successful Path B finalization

- `commit_hash <- HEAD`
- `status -> done`
- `next_step -> none`

## Task Planning Workflow

Normal tasks may go through an optional pre-make planning phase before code
changes begin.

### Task Planning Fields

- `commit_plan`: text field storing the plan drafted during `commit-plan`.
- Skips: Step-skipping can be configured to bypass `commit-plan` or `commit-plan-review`. Skipping `commit-plan` also bypasses `commit-plan-review`; the orchestrator also bypasses `commit-plan-review` when no `commit_plan` exists.

### Task Planning Flow

1. `commit-plan`: coder drafts an implementation plan and stores it via `task set <id> --commit-plan "<text>"`.
2. `commit-plan-review`: reviewer approves or rejects the plan.
3. On approval: task advances to `commit-make`.
4. On rejection: task returns to `commit-plan`. `review_round` is **not** incremented.

### Planning vs Commit Review

Task planning is distinct from commit review:

- `commit-plan`, `commit-plan-review`, `commit-make`, and `commit-review` are commit-task steps. Supertasks, pull request tasks, and other tasks must not use them.
- `commit-make-supertask` and `commit-review-supertask` are supertask-only steps.
- `pull-request-make` and `pull-request-review` are pull-request-only steps.
- `other-make` and `other-review` are other-task-only steps.
- Plan review decisions are determined by any new approval or rejection comment written during the review run, without filtering by `review_round`.
- `review_round` is only incremented by the commit review loop (`commit-review`), not by the planning loop.

### Task Planning State Transitions

#### On successful `commit-plan`

- `status -> ready`
- `next_step -> commit-plan-review` when plan review is not skipped
- `next_step -> commit-make` when plan review is skipped

#### On plan approval

- `status -> ready`
- `next_step -> commit-make`
- `last_review_decision -> none`

#### On plan rejection

- `status -> ready`
- `next_step -> commit-plan`
- `review_round` unchanged

#### On plan reviewer error

- `status -> blocked`
- `next_step -> none`

### Recovery

If the orchestrator starts and finds a task stuck in `running` at `commit-plan-review`,
it resets `next_step` to `commit-plan` (not `commit-plan-review`). This prevents stale
approval or rejection comments from a prior interrupted run from influencing the new run.

## Supertask Workflow

Supertasks are planning tasks with child execution.

### Supertask Rules

- A supertask never lands a commit.
- A supertask uses `commit-make-supertask` and `commit-review-supertask`.
- Child tasks must point at a supertask parent.
- Child tasks inherit the supertask branch.
- Child ordering is controlled by `sequence_index`.
- Siblings are renumbered at `100` intervals after insertion or reorder.

### Supertask Planning Flow

1. `commit-make-supertask`: coder writes the plan and creates child tasks
2. `commit-review-supertask`: reviewer approves or rejects the plan
3. On approval: supertask becomes `pending_subtasks`
4. Child tasks execute in sequence
5. When all children are `done`, the supertask becomes `done`

### Child Task Eligibility

A ready child task is eligible only when:

- its parent supertask is `pending_subtasks`
- no sibling with a lower `sequence_index` is still not `done`

### Parent and Child State Coupling

- Child tasks are created as `ready`.
- If a child becomes `blocked`, the parent supertask becomes `blocked`.
- If a blocked child is restored to `ready` and no other siblings remain blocked, the parent returns to `pending_subtasks`.
- When all children are `done`, the orchestrator marks the parent `done`.

## Orchestrator Contract

### Main Loop

The orchestrator:

1. initializes runtime state
2. recovers abandoned `running` tasks from previous instances
3. polls for the next eligible ready task
4. pins that task locally through its internal step loop
5. updates runtime and durable state as the task advances

### Ready Task Eligibility

A ready task is eligible for pickup when:

- its own workflow state allows the step to run
- child ordering allows it to run, when it belongs to a supertask
- either no task is currently `blocked`, or the task has `allow_when_blocked = true`
- if its branch is `master` or `main`, the repo has opted in with
  `ALLOW_TASKS_ON_MASTER`

Ready tasks skipped by the blocked-task gate stay `ready`. The orchestrator logs the
skip when the gate first applies to that task.

### Graceful Stop

Creating a file named `KANBAN_ORCHESTRATOR_STOP_AFTER_TASK` in the work repo root signals the orchestrator to stop after the current task finishes. The check happens at the top of the polling loop — after a task completes and before the next pickup.

When the file is detected:
- the orchestrator logs that it found and is acting on the file
- deletes the file
- exits the loop cleanly

The file is a one-shot signal: deletion is the acknowledgment, so the orchestrator will not stop again on the next run unless the file is recreated.

### Instance and Fleet Control

The primary runtime unit is a repo instance: one orchestrator process and one
dashboard process for the same git repo root. Start it from the work repo root:

```bash
"$ORCHESTRA_DIR/bin/ko-orchestrator"
```

When the work repo root is the same path as `$ORCHESTRA_DIR`, restart the
repo instance after pulling or changing Orchestra code so long-lived processes
do not keep using older in-memory code.

For multiple repos, `ko-fleet` reads a private flat path list from
`~/.config/orchestra/fleet.repos` by default. Each non-empty non-comment line
is one git repo root. `~` and environment variables are expanded, and display
names are derived from the basename of the resolved repo path.

```bash
"$ORCHESTRA_DIR/bin/ko-fleet" init
"$ORCHESTRA_DIR/bin/ko-fleet" add /path/to/work-repo
"$ORCHESTRA_DIR/bin/ko-fleet" precheck
"$ORCHESTRA_DIR/bin/ko-fleet" status
"$ORCHESTRA_DIR/bin/ko-fleet" start
"$ORCHESTRA_DIR/bin/ko-fleet" stop <repo-label>
"$ORCHESTRA_DIR/bin/ko-fleet" restart <repo-label>
"$ORCHESTRA_DIR/bin/ko-fleet" attach <repo-label>
"$ORCHESTRA_DIR/bin/ko-fleet" logs <repo-label>
"$ORCHESTRA_DIR/bin/ko-fleet" dashboard <repo-label>
```

`ko-fleet start`:
- starts the orchestrator/dashboard pair for every selected configured repo
- refuses duplicates when the repo singleton lock is already live
- refuses all selected starts when any selected repo is dirty or invalid
- keeps process-supervision details behind the fleet command

`ko-fleet stop`:
- stops fleet-owned terminal sessions for selected repos
- leaves externally managed instances alone and reports them

`ko-fleet restart` stops the selected fleet-owned instances, waits for the
repo-scoped orchestrator lock metadata to clear, and starts them again.
`ko-fleet attach`, `ko-fleet logs`, and `ko-fleet dashboard` select by repo
label/path and then use the selected repo's tmux session, repo-local
`.kanban-orchestra/orchestrator.log`, and repo-local dashboard metadata.
`ko-fleet dashboard-open` is an alias for `ko-fleet dashboard`.

The repo dashboard is the supported UI surface: it is read-only, repo-scoped,
and attached to the matching orchestrator instance. The old process-manager UI
and its heartbeat/request/response JSON files are removed, not compatibility
surfaces. Operator workflows should use `ko-orchestrator`, `ko-fleet`,
`ko-task`, and `ko-get-update`; active child process metadata is maintained
independently in `.kanban-orchestra/active-agent-processes.json`. The old
`BREAK` control is intentionally removed as a remote operator command: it killed
in-flight agent process groups and rewrote runtime state outside the
orchestrator's normal task loop. The supported migration is to stop or interrupt
the repo instance, inspect status with `ko-get-update`, record or unblock the
affected task with `ko-task`, and then restart the repo instance explicitly.

### Pinned Task Execution

Once a task is picked up, the orchestrator keeps processing that same task until
it reaches a non-ready state. This prevents other tasks from interleaving
between `commit-make` and `commit-review`.

### Agent ACK Gate

Before running any non-skipped agent-driven step, the orchestrator performs a
pre-flight ping for the assigned agent.

- Ping prompt: `This is a ping. Respond with ACK.`
- Any text response counts as a successful acknowledgment.
- A successful acknowledgment is cached per `(task_id, agent_name)` for the life of the current orchestrator process.
- On a cache miss with no response, the orchestrator keeps the task pinned, sets runtime `status_message` to a stalled wait message, and retries every `60` seconds.
- While the orchestrator is stalled on this gate, no other tasks run.

### Branch Handling

Before non-supertask steps:

- if the worktree is dirty at pickup for `commit-make`, the task is blocked immediately
- the orchestrator switches to the task branch
- if the branch does not exist, it may be created only when the base is unambiguous
- otherwise the task is blocked with a durable comment

### Review Round Limits

- Maximum review rounds: `5`
- Reaching the limit blocks the task instead of requeueing it

## Recovery and Failure Handling

### Startup Recovery

If the orchestrator starts and finds tasks still in `status=running`:

- for `commit-review` and `commit-review-supertask`, advance to the next review round and reset to `ready`
- for other steps, reset the task to `ready`
- leave a durable orchestrator comment describing the recovery

This prevents stale review comments from being mixed into a new round.

### Interrupt Recovery

If the orchestrator is interrupted during:

- `commit-review` or `commit-review-supertask`: requeue the same step at the next review round
- other steps: requeue the task at the same step

### Blocked `commit-make` With WIP

If `commit-make` fails after leaving local changes:

- the orchestrator stages the changes
- stashes them with `--include-untracked`
- records `stash_ref`
- leaves a durable comment
- marks the task `blocked`

If other ready tasks exist, the orchestrator continues only to tasks with
`allow_when_blocked = true`.

### Orchestrator-Level Errors

On unrecoverable internal exceptions:

- the current task becomes `blocked`
- an orchestrator comment is written
- runtime status becomes `error`

## Runtime and Dashboard Contract

The dashboard is read-only. It exists to answer operational questions quickly.
It is the per-repo dashboard started by the orchestrator, not the removed
process-manager UI.

Primary questions:

- Is the orchestrator alive?
- Is it idle, running, starting, stopped, in `hard-break`, stalled, stale, or in error?
- What task is active?
- What branch and step is active?
- What does recent task activity look like?
- What does current orchestrator output look like?
- What work is ready, blocked, or recently done?

### Runtime Semantics

- The orchestrator writes a singleton runtime row on startup.
- The orchestrator starts the matching dashboard for the same repo instance.
- Heartbeat updates every `10` seconds.
- A stale heartbeat indicates a dead or wedged orchestrator even if stored status still says `running`.
- Runtime `current_step` is normalized to one of the orchestrator step names
  (`commit-plan`, `commit-plan-review`, `commit-make`, `commit-review`,
  `commit-make-supertask`, `commit-review-supertask`,
  `pull-request-make`, `pull-request-review`, `other-make`, `other-review`) or `none`.
- Supertask detail is conveyed through `status_message`.
- The orchestrator appends stdout lines to `.kanban-orchestra/orchestrator.log`.
- A stalled agent ACK wait is surfaced through `status_message`.
- `starting`, `stopped`, and `hard-break` are terminal/operator-visible states
  and should not be mislabeled as stale solely because no orchestrator heartbeat
  is currently advancing.

### Minimum Dashboard Surface

The dashboard should expose:

- orchestrator health
- current task summary
- orchestrator output
- ready queue
- blocked tasks
- recent done tasks
- task detail view with comments and run log

### Data Sources

- `tasks`: durable workflow state
- `comments`: permanent decisions and notes
- `run_log`: task-scoped operational messages
- `orchestrator_runtime`: current liveness and focus
- `.kanban-orchestra/orchestrator.log`: persisted orchestrator output

## Prompts and Agent Contract

Prompt wording is implementation detail, but these surfaces are part of the contract:

- shared task context
- commit task make prompt
- commit task review prompt
- supertask make prompt
- supertask review prompt
- task plan prompt (`commit-plan`)
- task plan review prompt (`commit-plan-review`)

Required prompt inputs include:

Reviewer prompts must also surface:

## Human Operations

Humans mainly do four things:

- create and queue tasks
- inspect status, comments, and run logs
- unblock tasks with new guidance
- reopen landed tasks for follow-up work

## What A Human Does

The orchestrator owns normal workflow transitions, but humans are responsible for
intervention, triage, and requeue decisions.

### Create A New Task

Typical flow:

```bash
python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py add "Task title" --description "## Goal

What should happen" --branch my-branch
python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py set <task-id> --status ready
```

To create a task with the default `commit-make` starting step:

```bash
python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py add "Task title" --branch my-branch
python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py set <task-id> --status ready
```

Rules:

- A task must have a branch before it becomes `ready`.
- A task should have a meaningful `next_step` before it becomes `ready`.
- A task on `master` or `main` requires `ALLOW_TASKS_ON_MASTER` as a standalone
  line in `AGENTS.md`; otherwise use a feature branch.
- For child tasks, the branch comes from the parent supertask.
- Use `--allow-when-blocked` when a task should remain eligible while another task is `blocked`.

### Check What Is Going On

Useful commands:

```bash
python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py list
python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py show <task-id>
python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py show-comments <task-id>
python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py show-run-log <task-id>
sqlite3 kanban-orchestra.db "select status, current_task_id, current_step, current_branch, review_round, active_agents, status_message, last_heartbeat_at from orchestrator_runtime;"
```

### Unblock A Task

When a task is blocked because the agent needs a decision, context, or manual
repair:

```bash
cat <<'EOF' | python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py comment <task-id> --message-stdin --comment
Here is the missing decision or context
EOF
python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py set <task-id> --status ready --next-step commit-make
```

Rules:

- If blocked `commit-make` work was stashed, leave `stash_ref` intact so the next run can try to restore it.
- If the task should stay parked, set it to `status=none` and `next_step=none` instead of requeueing it.

### Recover From Branch Or Workspace Problems

Typical cases:

- If the repo is dirty before pickup, clean it up outside the task before requeueing.
- If the task branch was wrong or missing, fix the branch situation first, then set the task back to `ready`.
- If stashed WIP is no longer usable, clear `stash_ref`, leave a comment, and requeue the task for a rebuild.

### Reopen A Done Task

If landed work needs follow-up changes, reuse the same task:

```bash
cat <<'EOF' | python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py comment <task-id> --message-stdin --comment
Follow-up changes needed: ...
EOF
python3 "$ORCHESTRA_DIR"/kanban-orchestra/scripts/task.py set <task-id> --status ready --next-step commit-make --last-review-decision none
```

The task will re-enter the normal make/review/finalize loop and refresh
`commit_hash` when it lands again.

### Park Or Delete Work

- Park unfinished work: set `status=none` and `next_step=none`
- Delete abandoned work: `task delete <task-id>` only while the task is still `none`

Typical unblock actions:

- add a comment with missing context or a decision
- set a blocked task back to `ready`
- keep `stash_ref` intact if preserved WIP should be resumed
- park work by setting `status=none` and `next_step=none`

Typical reopen action for landed work:

- add guidance as a normal comment
- set the task back to `ready`
- set `next_step=commit-make`
- set `last_review_decision=none`

The next approved finalization updates `commit_hash` to the new landed commit.
